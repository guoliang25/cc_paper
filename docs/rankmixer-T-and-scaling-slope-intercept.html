Part I · RankMixer 里的 Token 数 $T$ 是不是拍的？

先厘清 RankMixer 论文里 "token 个数" 到底指什么。其实有 两组 容易混淆的 token 数：

符号含义论文典型值
$T$Feature Tokenization 后的 特征 token 个数100M 配置 $T=16$；1B 配置 $T=32$
$N_e$每层 FFN 的 expert 个数（Sparse-MoE）与 $T$ 无直接关系

我们讨论的是前者。具体构造方式（论文 2.2 节）：

$$x_i = \text{Proj}\big(e_\text{input}[\,d\cdot(i-1):d\cdot i\,]\big),\quad i=1,\ldots,T$$

把所有特征 embedding 拼成一条长向量 $e_\text{input}\in\mathbb{R}^{Td}$，均匀切 $T$ 段，每段做线性投影变成一个 $d$ 维 token。

$T$ 的五条约束

$T$ 是一个经验性工程超参，但它的可选空间被五个约束挤得非常窄：

约束 1 · 总参数 / FLOPs 预算

Dense 版本参数量公式：

$$\text{Param}\approx 2kLTd^2$$

主导项是 Per-token FFN 的 $T\cdot d^2$。$T$ 和 $d$ 同时作用。

约束 2 · Token Mixing 的整除性

无参数 Token Mixing 的 reshape：$\mathbb{R}^{T\times d}\to\mathbb{R}^{T\times T\times d/T}$，要求 $d\bmod T=0$，并且默认 $H=T$（head 数等于 token 数）。可选 $T$ 就被锁在 $\{8,16,32,64\}$ 这种 2 的幂。

约束 3 · Per-token FFN 的语义粒度

$T$ 对应 "把全部特征分成几个独立子空间做非线性变换"：

  • $T$ 太小 → user / item / context / cross 被迫共享 FFN，语义被稀释；
  • $T$ 太大 → 每 token 拿到的特征太少（$d$ 小），FFN 表达力不足。

约束 4 · GPU 硬件对齐

Tensor core 要求矩阵维度对齐 16/32/64 倍数。两个配置有意思的共同点：

配置$T$$d$head dim $=d/T$
RankMixer-100M1676848
RankMixer-1B32153648

→ head dim 锁在 48（tensor core 友好），$T$ 和 $d$ 联动变化。

约束 5 · Scaling Law 曲线平滑

论文 scaling 实验发现：扫 $T$、$d$、$L$ 三条轴 曲线几乎重合——$T$ 在合理范围内没有显著尖峰最优点，所以作者可以 "比较随意" 地取 16/32。

结论：$T$ 是有边界的"拍"——被五条约束挤到只剩 $\{8,16,32,64\}$ 这几个离散值，论文选 16/32 是 "在这几个候选里挑硬件对齐最舒服、语义粒度也合理的两个"。

Part II · $Td^2$ 怎么变：参数量辨析

上一轮我说过一句 "$T$ 越大、每 token 维度 $d$ 越小，参数量反而越小"——这个说法省略了关键前提，在默认视角下是错的。本节把账算清楚。

参数量主导项：Per-token FFN 的 $T\cdot d^2$

RankMixer Block 里 Token Mixing 无参数（reshape+转置），也 没有 QKV（用 Token Mixing 代替了 self-attn），所以参数只来自：

  • Per-token FFN：$L\cdot T\cdot 2kd^2$ —— 主导项
  • 入口投影：$E\cdot d$，在 $E=Td$ 时等于 $Td^2$

$T$ 变大时，$Td^2$ 怎么变？取决于"你固定了什么"

情境 固定的量 $Td^2$ 随 $T$ 的走向 走向 谁对？
A · "切蛋糕"视角 总 embedding 预算 $E=Td$（即 $d=E/T$） $Td^2 = E^2/T$ ↓ 随 $1/T$ 下降 我上一轮说的
B · "堆 FFN"视角 每 token 维度 $d$ $Td^2 \propto T$ ↑ 线性上升 你的直觉
C · 只扩 $d$ $T$ 固定 $Td^2 \propto d^2$ ↑ 二次上升 widen 轴 scaling
D · 论文实际 两者同步放大 $T:16\to32,\ d:768\to1536$，$Td^2$ 放大 8× ↑↑ 快速放大 100M → 1B

直觉对照

情境直觉
A · 固定 $E$ "一块蛋糕切成更多份，每份缩小得比份数增加快"——$d$ 下降的 $d^2$ 效应比 $T$ 上升的线性效应更强。
B · 固定 $d$ "每份大小不变，份数越多总量越大"——Per-token FFN 一套套堆上去。
D · 同时扩 "又加份数又加每份大小"——参数量爆炸式增长。

为什么 "固定 $E$" 在工业场景是常见前提？

  1. 原始特征 embedding 的总维度 $E$ 由业务决定（比如 500 个字段 × 16 维 = 8000），不是自由变量；
  2. Feature Tokenization 的设计把 $e_\text{input}\in\mathbb{R}^{Td}$ 均匀切 $T$ 份——$T$ 和 $d$ 在入口就是绑定的；
  3. 但这个前提可以被打破——论文 100M → 1B 时 $E$ 从 12288 增到 49152，显然 $E$ 本身也扩了。
两种视角都对，关键看你固定哪个变量。工业上比较 token 切分策略时用 A（固定 $E$）；做 scaling 实验时用 D（两者同扩）。原来的说法应该加上 "在总 embedding 预算给定的前提下" 这个限定词。

Part III · Scaling Law 的斜率与截距

为什么 scaling law 可以画成一条直线？

工业推荐 scaling 论文通常拟合：

$$y \approx \alpha\cdot x^{\beta} + \gamma$$

其中 $x$ 是算力类变量（参数量、FLOPs、序列长度），$y$ 是性能指标（AUC / ΔAUC / LogLoss）。画成 x 轴 log、y 轴线性 后，曲线近似直线：

$$y \approx \beta\cdot\log x + (\text{const})$$

一旦变直线，就有两个传统几何量：斜率和截距。

斜率（slope）= 算力的兑换率

定义：$x$ 每翻倍，$y$ 涨多少。数学上 ≈ scaling law 里的指数 $\beta$。
工业含义：每多烧一倍算力能换回多少性能——架构的 scaling 效率 / 远期潜力。
ΔUAUC (%) ↑ ╱ OneTrans（陡） │ ╱ │ ╱ ← 每翻倍 FLOPs，OneTrans 多涨一截 │ ╱────── RankMixer（平） │ ╱ │ ╱ │____╱___________________→ log(FLOPs) 10² 10³ 10⁴
  • 斜率陡：加算力值得——"远期潜力大"；
  • 斜率平：加算力没用——"架构饱和了"。

截距（intercept）= 起跑线

定义：拟合直线在 $\log x=0$（曲线左端）的 $y$ 值——直线在 y 轴上的抬升高度。
工业含义：给最小算力时，架构能到什么性能——小模型的底子好不好。
ΔAUC ↑ ╱ MixFormer（截距高 + 斜率持平） │ 截距 ───► ╱╱╱ │ ╱ │ ╱ ────── SOTA 序列模型 │ ╱ │ ╱ │____╱_______________________→ log(Sequence Length) 512 1k 2k 5k 10k
  • 截距高："一开始就领先"；
  • 截距低："起点低"，但靠斜率可能追回来。

斜率 × 截距的四种组合

斜率截距含义典型案例
高高双优——起点高、上限也高Chinchilla 风格新一代架构
高低后发制人——小时候不行，越 scale 越强Transformer 早期 vs LSTM
低高早期赢家——起点高但加算力不涨饱和的老架构
低低全面落后Baseline
关键区分：
  • "斜率更高" = 未来更值得投资（扩上去差距会扩大）
  • "截距更高" = 现在就比你强（但扩上去不一定继续甩开）
两者独立，语义完全不同。

Part IV · 四大架构的 scaling 曲线实例

把对比文档里的几条 scaling 描述，配上原始论文的图看一遍。

① RankMixer · 比 Wukong / HiFormer / DHEN 斜率更陡

翻译成大白话： "同样把参数从 100M 扩到 1B，RankMixer 的涨幅比 Wukong / HiFormer 大——远期更值得投资。"

② OneTrans · 斜率显著 > RankMixer

OneTrans 论文（Fig.3(b)）把自己和 RankMixer 的 ΔUAUC-vs-log(FLOPs) 画在一张图里，OneTrans 那条线更陡。论文的解读是：

RankMixer-centric scaling lacks a unified backbone; its MoE-based expansion predominantly widens the FFN hidden dimension. OneTrans, offering a unified Transformer backbone, scales more parameter- and compute-efficiently.

直白翻译："RankMixer 扩上去主要靠增加 MoE FFN 的宽度，没有统一 backbone；OneTrans 有统一 backbone，所以 scaling 效率更高。"

③ MixFormer · 斜率与 SOTA 序列模型持平，截距更高

翻译成大白话： "MixFormer 在各种长度档位都比对手高一个固定幅度；再拉长也不会进一步拉开——一开始就胜，扩上去也不掉。"

④ LONGER · 三轴各自独立 power-law 拟合

LONGER 论文（Fig.4 + Fig.5）把 sequence length / params / FLOPs 三条轴 分别 拟合 $y=\alpha x^\beta+\gamma$，都得到 $R^2>0.96$——意味着：

  • 长度从 300 → 5k，AUC / LogLoss 随 length 呈 power-law 改善；
  • 固定 2 层、宽度扫描，AUC vs params 强 power-law（$R^2=0.987$）；
  • 固定 $d=32$、扫 layers + length，AUC vs FLOPs 强 power-law（$R^2=0.967$）。

可惜 LONGER 论文没有跟其它架构在同一张图里对比斜率。

Part V · 一句话总结

  1. $T$ 不是纯拍的：被 FLOPs 预算、Token Mixing 整除性、Per-token FFN 粒度、GPU 硬件对齐、scaling law 曲线五条约束挤到 $\{8,16,32,64\}$ 几个离散值，论文选 16/32 是 head dim 锁到 48 的硬件友好选择。
  2. $Td^2$ 怎么变取决于你固定什么：固定 $E=Td$ → $1/T$ 下降；固定 $d$ → 线性上升；论文实际两者同扩。
  3. Scaling law 的斜率 = 兑换率（未来潜力）；截距 = 起跑线（当下胜负）。
  4. 四大架构的 scaling 定位：
    • RankMixer 比前代斜率陡（FI 领域内的升级）；
    • OneTrans 斜率 > RankMixer（统一 backbone 带来的"未来价值"）；
    • MixFormer 斜率持平、截距更高（现在就比你强）；
    • LONGER 三轴均强 power-law，但没画跨架构对比。
