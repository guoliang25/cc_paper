Part I · NLP 里 Adapter 具体是怎么做的

1. Adapter 模块本体：一个最浅的 bottleneck

数学定义（$h$ 是 sub-layer 输出，$d$ 是 hidden dim，$m$ 是 bottleneck dim，$m \ll d$）：

$$\text{Adapter}(h) = h + W_\text{up}\,\sigma(W_\text{down}\,h)$$

张量形状流（BERT-base $d=768$、$m=64$）：

步骤输入操作输出参数量
Input$[B, L, d]$—$[B, L, d]$—
Down-project$[B, L, d]$$h W_\text{down}^T + b$$[B, L, m]$$dm + m$
Nonlinearity$[B, L, m]$GeLU$[B, L, m]$0
Up-project$[B, L, m]$$\cdot W_\text{up}^T + b$$[B, L, d]$$md + d$
Skip-connection$[B, L, d]$$+ h$$[B, L, d]$0

总参数 $2md + d + m \approx 100\text{K}$，对比一个 FFN 的 ~4.7M，Adapter 只占 ~2%。

关键细节：

  • Adapter 插在 sub-layer 输出之后、残差相加之前（接管"sub-layer 的增量信号"）
  • 每层有两个 Adapter（MHA 后一个 + FFN 后一个），BERT-base 12 层 → 共 24 个 Adapter

4. Near-Identity 初始化（最重要但最容易被忽略的细节）

$W_\text{down}, W_\text{up}$ 用 zero-mean Gaussian 初始化，$\text{std}=10^{-2}$：

$$W_\text{down}, W_\text{up} \sim \mathcal{N}(0, 10^{-4})\text{ truncated}$$

训练第 0 步时 $W_\text{up}\sigma(W_\text{down}h) \approx 0$，整个 Adapter 输出 $\approx h$——恒等变换。所以：

  • 带 Adapter 的模型在 step 0 时性能 = 原始冻结 BERT
  • Adapter 在训练中 逐步"激活"，不会扰坏预训练权重
  • 消融显示 $\text{std} \le 10^{-2}$ 都稳，$>10^{-2}$ 开始不稳

5. 训练时谁被训、谁被冻

组件状态参数占比
MHA 的 $W_Q, W_K, W_V, W_O$冻结大头
FFN 权重冻结大头
Token / Position Embedding冻结中
Adapter 的 $W_\text{down}, W_\text{up}, b$训练~3%
每层 LayerNorm $\gamma, \beta$训练<0.1%
任务分类头训练<0.1%

6. 10 行 PyTorch 实现

class Adapter(nn.Module):
    def __init__(self, d, m):
        super().__init__()
        self.down = nn.Linear(d, m)
        self.up   = nn.Linear(m, d)
        nn.init.normal_(self.down.weight, std=1e-2)
        nn.init.normal_(self.up.weight,   std=1e-2)   # 近零初始化
        nn.init.zeros_(self.down.bias)
        nn.init.zeros_(self.up.bias)
        self.act = nn.GELU()

    def forward(self, h):
        return h + self.up(self.act(self.down(h)))    # skip-connection

路径 1 · 多任务 Ranking 里的任务专属 Adapter

现状问题：CTR / CVR / 停留时长 / 完播率 / 点赞率通常共享 backbone（比如 RankMixer），然后每个任务一个 Tower。但 Tower 容量有限，用大 Tower 又会让共享部分的梯度被稀释。

flowchart LR RM["Shared RankMixer
(冻结或低 LR)"] RM --> AC[Adapter_CTR] --> TC[Tower_CTR] --> LC[CTR logit] RM --> AV[Adapter_CVR] --> TV[Tower_CVR] --> LV[CVR logit] RM --> AW[Adapter_WatchTime] --> TW[Tower_WT] --> RW[回归] RM --> AL[Adapter_Like] --> TL[Tower_Like] --> LL[Like logit] style AC fill:#fff3e0,stroke:#fb8c00 style AV fill:#fff3e0,stroke:#fb8c00 style AW fill:#fff3e0,stroke:#fb8c00 style AL fill:#fff3e0,stroke:#fb8c00 style RM fill:#e3f2fd,stroke:#1976d2,stroke-width:2px

收益：

  • 新增任务不影响旧任务（迁移保证）
  • 任务专属参数 10–100× 于单层 Tower
  • 训练数据少的任务（评论、转发）可共享主任务的 backbone

路径 2 · 多场景 Ranking 里的场景 Adapter

现状问题：Feeds / Mall / 搜索 / 直播 / 广告 每个场景独立训模型——数据重复、算力浪费、冷启动难。OneTrans / MixFormer 已经在用 "unified backbone" 解决前两个问题，但场景差异如何体现？

flowchart TB I[同一请求] I --> F[场景 Feeds] I --> M[场景 Mall] I --> S[场景 搜索] I --> L[场景 直播] F --> BK["Shared Backbone
(OneTrans / MixFormer)"] M --> BK S --> BK L --> BK BK --> AF[Adapter_Feeds] BK --> AM[Adapter_Mall] BK --> AS[Adapter_Search] BK --> AL[Adapter_Live] AF --> TF[Tower & logit] AM --> TM[Tower & logit] AS --> TS[Tower & logit] AL --> TL[Tower & logit] style AF fill:#fff3e0,stroke:#fb8c00 style AM fill:#fff3e0,stroke:#fb8c00 style AS fill:#fff3e0,stroke:#fb8c00 style AL fill:#fff3e0,stroke:#fb8c00 style BK fill:#e3f2fd,stroke:#1976d2,stroke-width:2px

关键点：Adapter 插在 backbone 的每一层（类似 NLP 的做法），让场景差异能在深层体现；新场景冷启动用"相似场景 Adapter"初始化。对比 PEPNet / STAR：Adapter 方案在"可扩展性和任务隔离"上更干净。

路径 3 · LLM-based Recommender 里的任务 Adapter（最贴近 NLP 原意）

这是 Adapter 最"像它本来长的样子"的推荐场景：

(1B-7B, 冻结)"] UL --> AU[Adapter_User 任务/场景] AU --> UE[user representation] UE --> TASK[CTR / CVR / ...] style IL fill:#e3f2fd,stroke:#1976d2,stroke-width:2px style UL fill:#e3f2fd,stroke:#1976d2,stroke-width:2px style AI fill:#fff3e0,stroke:#fb8c00 style AU fill:#fff3e0,stroke:#fb8c00

工程意义：HLLM 主干 7B 冻结（只存一份权重），每个推荐任务/行业/场景只训一组 Adapter（几 M 参数）。让 LLM-based Recommender 真的能下沉到百万级场景。

Part III · 多模态序列 × RankMixer 的 Adapter 方案（主推）

这是最贴合 "RankMixer 基线 + 多模态用户序列" 真实工业场景的方案。核心思想：把多模态序列建模做成一个带 Cross-Attention 的 Adapter，挂到冻结的 RankMixer 上——同时享受 Adapter 的"零风险增量"和 Cross-Attention 对长序列的聚合能力。

公式化定义

设 RankMixer 第 $\ell$ 层输出为 $H^{(\ell)} \in \mathbb{R}^{B \times T \times d_\text{rank}}$，多模态序列编码为 $M \in \mathbb{R}^{B \times L \times d_\text{mm}}$。MM-Adapter 的计算流：

$$Z = H^{(\ell)} W_\text{down} \in \mathbb{R}^{B \times T \times m}$$ $$Z' = \text{MultiHeadAttn}(Q=Z,\ K=M W_K,\ V=M W_V)$$ $$H^{(\ell+1)} = H^{(\ell)} + Z' W_\text{up}$$

其中 $W_\text{down} \in \mathbb{R}^{d_\text{rank} \times m}$、$W_\text{up} \in \mathbb{R}^{m \times d_\text{rank}}$，遵循 近零初始化。

PyTorch 实现

class MultiModalAdapter(nn.Module):
    """
    Cross-Attention Adapter for injecting multimodal sequence info
    into a frozen RankMixer backbone.
    """
    def __init__(self, d_rank, d_mm, m=64, n_heads=4):
        super().__init__()
        self.down = nn.Linear(d_rank, m)
        self.mm_attn = nn.MultiheadAttention(
            embed_dim=m, num_heads=n_heads,
            kdim=d_mm, vdim=d_mm, batch_first=True
        )
        self.up = nn.Linear(m, d_rank)

        # Near-identity init: Adapter 启动时输出 ~ 0，不扰动 RankMixer
        nn.init.normal_(self.down.weight, std=1e-2)
        nn.init.normal_(self.up.weight,   std=1e-2)
        nn.init.zeros_(self.down.bias)
        nn.init.zeros_(self.up.bias)

    def forward(self, h_rank, mm_seq_kv, mm_key_padding_mask=None):
        """
        h_rank:       [B, T, d_rank]   RankMixer 中间层输出 (T 个特征 token)
        mm_seq_kv:    [B, L, d_mm]     多模态序列编码 (L 个行为 token)
        mm_key_padding_mask: [B, L]    可选，屏蔽 padding
        """
        z = self.down(h_rank)                    # [B, T, m]
        z_out, _ = self.mm_attn(
            query=z, key=mm_seq_kv, value=mm_seq_kv,
            key_padding_mask=mm_key_padding_mask
        )                                        # [B, T, m]
        return h_rank + self.up(z_out)           # residual

Part IV · 落地 checklist

NLP Adapter 原版要点

  • ☐ Bottleneck $m$：取 $d/16 \sim d/8$（$d=768$ 时选 $m=48\sim 96$）
  • ☐ 非线性：GeLU（或 ReLU）
  • ☐ Skip-connection：内层 Adapter 自己一个，外层残差流保留
  • ☐ 初始化：$W_\text{down}, W_\text{up}$ zero-mean Gaussian std=$10^{-2}$，bias=0
  • ☐ 冻谁训谁：主干全冻，只训 Adapter + LN + 任务头
  • ☐ 插位：每个"主处理 block"的输出后各一个
  • ☐ 学习率：Adapter LR 比主干 fine-tune 的 LR 高 ~10×

推荐系统 Adapter 工程 checklist

  • ☐ 先跑 baseline，确认 RankMixer 主干指标稳定
  • ☐ Stage 2 冻主干训 Adapter——快、便宜，能立即回答"Adapter 值不值"
  • ☐ 分长尾桶看指标——Adapter 对冷启动/新场景的增益应显著大于热门场景
  • ☐ 对比 Tower-only baseline——Adapter 必须超过"只扩大 Tower"才算有价值
  • ☐ 监控 Adapter 权重 norm——一直接近 0 = 没学到；爆炸 = 初始化太大
  • ☐ 保存/加载策略：Adapter 每任务独立存（几 MB），像配置文件一样管理
  • ☐ A/B 关注长尾：Adapter 版作 treatment，原 baseline 作 control，重点看冷启动 / 新用户 / 长尾 item

Part V · 一句话总结

NLP 里 Adapter = 在 Transformer 每个 sub-layer 后插一个 down(d→m) → GeLU → up(m→d) + skip 的极简 bottleneck，近零初始化保证启动时恒等，冻主干只训 Adapter（+ LN + 任务头），3% 参数拿到 99.5% 的性能。

迁移到推荐系统有四条路：多任务 Adapter（最简单）、多场景 Adapter（最通用）、LLM-based Recommender 里的任务 Adapter（最贴近 NLP 原意）、多模态序列作为 Cross-Attention Adapter 挂到 RankMixer（最贴合当前场景）。

对"RankMixer 基线 + 多模态序列"的场景，推荐第四条方案——先冻 RankMixer，把多模态信号做成一个带 Cross-Attention 的 Adapter 挂进去，分三阶段训练。这样既享受 Adapter 的"零风险增量"，又享受 Cross-Attention 对长序列的聚合能力，且完全可回滚。
